## Component Breakdown

### Edge Load Balancer (`cluster/cluster_load_balancer.py`)

- **Role:** Transparent message broker between all clients and all workers.
- **Socket type:** `zmq.ROUTER` (frontend) + `zmq.DEALER` (backend).
- **Forwarding mechanism:** `zmq.proxy(frontend, backend)` — a kernel-level forwarding loop. No Python GIL involvement. Zero data copying beyond the ZMQ kernel buffer.
- **High-water marks:** Both sockets set to `RCVHWM=1000, SNDHWM=1000`. When a worker is slow or a client floods, the kernel drops the oldest frames from the buffer instead of growing memory unboundedly.

### Compute Worker Node (`cluster/cluster_worker_node.py`)

- **Socket type:** `zmq.DEALER` — automatically registers with the proxy when `connect()` is called.
- **Receive loop:** `socket.recv_multipart()` → decrypt (AES-256-GCM) → parse → `BCIEngineOrchestrator.step()` → encrypt → `socket.send_multipart([identity, b"", encrypted])`.
- **Dynamic pipeline allocation:** Per-user `BCIEngineOrchestrator` instances are created on first sight via `initialize_lazy=True`. No pre-provisioning needed.
- **Fault isolation:** Exceptions in any single frame are caught and dropped. A malformed packet from one user cannot affect other users' pipelines.

### Client (Unity / Headset)

- **Outbound:** Sends encrypted JSON payload frames.
- **Connection target:** The load balancer's public-facing port (e.g., `tcp://<balancer-ip>:5557`).
- **Response:** Receives `(action_id, confidence, p_error)` routed back through the same proxy.

## Horizontal Scaling

Workers are stateless with respect to routing (state is per-user, not per-worker). Adding workers:

```bash
# On each new machine / container:
python -m cluster.cluster_worker_node \
    --balancer tcp://<balancer-ip>:5558 \
    --shared-key $(cat /etc/bci-key.hex)
```

The load balancer distributes incoming frames across all connected DEALER workers automatically via round-robin. No configuration change needed on the balancer or clients.
