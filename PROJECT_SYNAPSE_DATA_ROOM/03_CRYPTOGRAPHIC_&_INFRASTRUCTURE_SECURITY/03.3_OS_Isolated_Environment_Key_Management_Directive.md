# OS-Isolated Environment Key Management Directive — Project Synapse

## Policy

The 256-bit AES-GCM symmetric key used by `SecureBCIGateway` MUST be supplied through the `BCI_SHARED_KEY` environment variable. The runtime will refuse to start if this variable is unset, empty, or contains a string that does not decode to exactly 32 bytes (64 hex characters).

## Approved Injection Methods

### Linux / macOS (systemd / launchd / shell)

```bash
# Generate (one-time, store in a secrets manager):
export BCI_SHARED_KEY=$(openssl rand -hex 32)

# systemd service unit:
# Environment="BCI_SHARED_KEY=<value>"
# Or via systemd-creds:
# EnvironmentFile=/etc/bci-key.env  (mode 600)
```

### Windows (Docker / PowerShell)

```powershell
# Per-session:
$env:BCI_SHARED_KEY = "<64-hex-chars>"

# Docker Desktop:
# docker compose run -e BCI_SHARED_KEY=<value> ...
```

### Docker Compose

```yaml
services:
  bci-runtime:
    environment:
      - BCI_SHARED_KEY=${BCI_SHARED_KEY:?error}
```

The `:?error` syntax causes `docker compose up` to fail immediately if `BCI_SHARED_KEY` is not set in the host shell.

## Prohibited Practices

- ❌ Hardcoding the key in `daemon_config.json`
- ❌ Storing the key in a `.env` file committed to version control
- ❌ Passing the key as a command-line argument (visible in `/proc/*/cmdline`)
- ❌ Writing the key to application logs or stdout
- ❌ Embedding the key in the Docker image at build time

## Rotation Procedure

1. Generate a new key: `openssl rand -hex 32`
2. Update the environment variable on the host.
3. Restart the container/process: `docker compose restart`
4. All in-flight connections are dropped. Clients must reconnect with the new key.
